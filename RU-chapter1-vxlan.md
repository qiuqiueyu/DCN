III. Архитектура spine-leaf

Описание

POD — это базовая единица данных центра, то есть архитектура spine-leaf, также известная как IP Fabric. Разные POD выполняют разные сервисы: на каждом leaf активируется loopback, а маршруты loopback имеют несколько следующих прыжков ECMP.

Архитектура spine-leaf также называется архитектурой Clos, архитектурой "хребет-лист" или архитектурой "толстого дерева".

3.1 Ключевые детали архитектуры

- Все линии между spine и leaf являются интерфейсами L3 с IP-адресами.

- Классификация ролей leaf:
        

  - server-leaf: подключение серверов снизу

  - border-leaf: подключение устройств PE снизу

  - service-leaf: подключение межсетевых экранов и LB (VAS) снизу

- Роль spine: промежуточное устройство для пересылки, легко масштабируется.

3.2 Отличие от традиционной трехуровневой архитектуры LAN

Недостаток традиционной трехуровневой архитектуры LAN: для избежания циклов в сети L2 необходимо использовать STP, что приводит к блокировке линий, низкой утилизации и медленной сходимости STP по сравнению с IP.

Преимущества архитектуры spine-leaf: сеть IP может расширяться бесконечно, легко масштабируется, одиночная неисправность spine не влияет на общее обслуживание (только часть сервисов совершает обход).

Интегрированная архитектура: в будущем в некоторых сценариях spine также будет выполнять роли border-leaf и service-leaf, что называется интегрированной архитектурой.

3.4 Ключевые проблемы архитектуры и решения

Проблема: В данных центрах обычно большой объем восточно-западного трафика; каждый член vlanxx, в отличие от традиционной сети, не имеет фиксированного физического положения, может мигрировать и разбросаться по нескольким leaf; сеть IP spine-leaf прерывает vlan, что приводит к невозможности взаимодействия членов vlan через разные leaf.

Решение: Протокол VXLAN — технология туннелирования, заключающая vlan в IP-слой, относящаяся к сети наложения (overlay). Наложение новой сети на существующую сеть underlay (архитектура spine-leaf) позволяет实现 взаимодействие ВМ одного vlan под leaf через исходную сеть L3.

IV. VXLAN и BD

4.1 Принцип

- VXLAN — это проявление сетевой виртуализации, передача данных L2 по сети IP underlay с использованием туннелирования.

- Структура пакета: заголовок VXLAN размером 8 байтов, включая 3B VNI (2^24 номера VNI, около 16 миллионов), целевой UDP-порт 4789, исходный порт генерируется случайно (для полного использования путей ECMP).

- 说明 о накладных расходах: после упаковки исходного эфирного кадра в VXLAN накладные расходы увеличатся на 50B, поэтому IP MTU необходимо увеличить с 1500 до 1550.

VXLAN — это технология сетевой виртуализации MAC in UDP, упаковка пакета заключается в добавление упаковки UDP и заголовка VXLAN перед исходным эфирным пакетом. Конкретный формат пакета如下:

![Формат пакета VXLAN](images/vxlan-packet.png)

Расбор структуры пакета:

- Внешний слой: заголовок IP, заголовок UDP

- Заголовок VXLAN: 8 бит VXLAN Flags (фиксировано 00001000), 24 бита VNI, 24 бита Reserved, 8 бит Reserved

- Внутренний слой: исходный эфирный кадр (MAC DA, MAC SA, тег 802.1Q, тип Ethernet, данные Payload)

4.4 Связь BD с VLAN и VNI

- BD (Bridge Domain) выполняет функцию соединения, связывая VLAN с VXLAN, BD ID действует локально.

- Устройства ZTE: использование VPLS для привязки некоторых VLAN по dot1q.

- Ключевая логика: один BD соответствует одному VNI, но разные VLAN могут быть привязаны к одному BD для实现 взаимодействия между разными VLAN.

- 说明 интерфейса: не нужно создавать vlanif, использовать интерфейс vbdif в качестве интерфейса L3 данного BD для взаимодействия между разными BD; устройства ZTE используют irbxxx вместо vbdif, привязка через bridge-mapping.

4.7 Конфигурация изоляции VRF

Ключевое требование: таблица публичных маршрутов spine-leaf (глобальная таблица маршрутов) содержит только маршруты underlay (loopback spine и leaf, маршруты портов), маршруты overlay клиента необходимо изолировать, поэтому нужно создать отдельный VRF.

  vni-label 6017
  route-target export 1:6017
  route-target import 1:6017

  ${{address−familyipv6}}$
$

4.8 Отличие между NVE и VTEP

- NVE: точка начала туннеля устройства (общая точка начала туннеля).

- VTEP: конкретно указывает на точку начала туннеля VXLAN, использует IP loopback в глобальной таблице маршрутов в качестве идентификатора, обычно публикуется в spine через OSPF (spine относится к area 0).

- Вопрос: Можно ли адрес VTEP публиковать через BGP? (Использование BGP вместо OSPF для глобальной таблицы маршрутов возможно)

4.9 Причина случайного исходного UDP-порта VXLAN

Для максимальной случайности маршрутизация основывается на потоке (пятиэлементный кортеж: исходный и целевой IP, исходный порт, целевой порт, номер протокола), другие атрибуты неизменны, единственный изменяемый атрибут — исходный порт. Номер исходного UDP-порта генерируется в зависимости от нагрузки, что позволяет实现 хэширование, благодаря чему leaf при выборе следующего прыжка в уровне underlay выбирает более случайный путь и полностью использует пути ECMP.

6.3 Логика пересылки кадров BUM

- Униicast-кадры: пересылка по таблице MAC.

- Кадры BUM (не найдено в таблице MAC): пересылка по принципу "копирования на исходной стороне", текущий leaf копирует кадр и отправляет его каждому другому leaf, имитируя широковещательное поведение с помощью нескольких униicast, что создает большой накладной расход на устройство.

- Механизм защиты от циклов: горизонтальное разделение, то есть кадры BUM, полученные из VTEP, не будут пересылаться другим VTEP.

VII. Режимы пересылки VXLAN (симметричный vs асимметричный)

![Сравнение режимов пересылки](images/forward-mode.png)

（Примечание: Локально необходимо создать папку `images` и разместить соответствующее изображение с именем forward-mode.png, GitHub может отобразить его напрямую）

В вышеуказанной схеме красный цвет — асимметричная пересылка, зеленый — симметричная пересылка.

7.1 ZTE (асимметричная пересылка)

Шлюз leaf1 после проверки таблицы VRF 1 напрямую инжектирует ее в локальную таблицу MAC VPLS 10, упаковывая L2 VNI таблицы MAC 10; при обратном пути шлюз leaf2 проверяет таблицу маршрутов VRF1 и инжектирует ее в локальную таблицу MAC VPLS 20, упаковывая L2 VNI таблицы MAC 20. Поскольку VNI в оба направления разные, это называется асимметричной пересылкой.

7.2 Huawei (симметричная пересылка)

Шлюз leaf1 после проверки таблицы VRF 1, соответствующий следующий прыжок — туннель VXLAN, в этом случае упаковывается VNI таблицы VRF1 (то есть L3 VNI); после достижения leaf2 проверяется таблица VRF1 leaf2 и выполняется пересылка. Это эквивалентно двум маршрутизациям, в оба направления упаковывается один и тот же L3 VNI, поэтому это называется симметричной пересылкой.

IX. Централизованный шлюз

Описание централизованного шлюза

Централизованный шлюз: шлюз находится в spine, в этом случае leaf называется шлюзом VXLAN L2, а spine — шлюзом VXLAN L3.Недостаток: одиночная точка отказа, требуется двойная активность или четверная активность; большой накладной расход на spine, VNI имеет 16 миллионов, но количество интерфейсов vbdif не может превышать 4K (требования параметров продукта).Решение: vbdif от 1 до 4K размещаются на spine1-2, vbdif от 4K+1 до 8K — на spine3-4, vbdif в одном VRF должны быть размещены на одной группе spine, таким образом решается проблема большого накладного расхода; проблема неоптимального пути возникает при пересечении подсетей под одним leaf.

9.1 Классификация шлюзов

- Шлюз L2: устройство, реализующее связь vlan---BD----vxlan.

- Шлюз L3: интерфейс L3 интерфейса vbdif (irb), не может быть привязан к public VRF (глобальной таблице маршрутов), глобальная таблица маршрутов используется только для underlay.

9.2 Реализация мультиактивности spine

Адреса loopback TEP, указанные для spine1 и spine2, одинаковы; в глобальной таблице маршрутов leaf можно видеть два следующих прыжка к этому адресу loopback (например, 10.10.10.4), которые направлены на два интерфейса (например, 192.168.1.1 и 192.168.4.1).

9.3 Распространенные проблемы двойного активного шлюза

Проблема: Централизованный двойной активный шлюз (одинаковая конфигурация на двух spine), при пинге ВМ из шлюза ответы периодически разрываются.

Причина: Прямой путь проходит через spine1 для проверки маршрутов, обратный путь проходит через spine2 для проверки маршрутов и пересылки; если у шлюза нет синхронизации таблиц ARP, обратный путь может привести к потере первого пакета из-за отсутствия записей в таблице.

Решение: Таблицы ARP одного VRF на spine1 и spine2 необходимо синхронизировать.
